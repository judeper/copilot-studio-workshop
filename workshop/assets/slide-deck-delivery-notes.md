## Transition bridges

### Slide 5 - Into setup

- Move from workshop value to execution.
- Tell the room that every later module depends on correct account, environment, and browser state.
- Ask participants to surface blockers immediately instead of trying to catch up silently.

### Slide 41 - From build to publish

- Summarize what the room has already proven: grounded answers, structured inputs, flows, and event behavior.
- Shift the language from `can it work?` to `can we own, support, and publish it?`
- If the day is running long, shorten channel commentary before cutting the licensing and ROI close.

### Slide 46 - Day 2 restart

- Reframe Day 2 as the enterprise extension of Day 1, not a new scenario with new rules.
- Carry forward the Day 1 habits: scoped instructions, grounded sources, evidence in **Activity**, and disciplined naming.
- Tell the room that every Day 2 module increases the production-readiness pressure.

### Slide 73 - Into MCP

- Introduce MCP as governed extensibility, not open-ended plugin sprawl.
- Keep the message on supported setup, auditable tool scope, and admin trust.
- If the wizard or catalog is unavailable, switch to demo mode quickly and protect the concept.

### Slides 84-85 - Optional developer branch

- Keep the main room on evaluation, ROI, and rollout actions.
- Direct only caught-up developers to the VS Code branch.
- Reinforce that the core workshop is complete without Lab 25.

## Pre-lab facilitator guidance by module

Modules below have thin concept coverage relative to their lab time. The facilitator should compensate with the whiteboard activities, live demos, and orientation walkthroughs described here. Each item is designed to run during the slide portion of the module, before participants begin the hands-on lab.

### Module 03 — Reuse Patterns (slides 21-23, Labs 05; optional Lab 03 Agent Builder section)

- **Pre-check the room for Microsoft 365 Copilot licenses.** Before slide 21a, ask the room "Who has a Microsoft 365 Copilot license assigned to their workshop account?" and cross-reference the output of `Invoke-WorkshopPrereqCheck.ps1`. If most of the room is licensed, the optional Agent Builder walkthrough at the end of Lab 03 is hands-on for everyone. If fewer than half are licensed, run the walkthrough as a single facilitator demo from your own licensed tenant and keep participants on the Copilot Studio path.
- **Demo: Agent Builder in Microsoft 365 Copilot.** Open `m365.cloud.microsoft/copilot`, select **Agents → Build an agent**, and paste the Lab 03 sample description ("Create an agent that helps Woodgrove Bank customers find the right loan product based on their goals…"). Show participants how Agent Builder generates instructions, starter prompts, and optional knowledge attachment in under a minute. Test one prompt and call out the surface differences (no Activity map, no Power Fx, no environment picker) so the contrast with Copilot Studio is visible, not just described.
- **Frame the choice with the contrast table on slide 21a.** Walk the five rows (audience reach, authoring depth, governance, knowledge sources, best-for) and tie each row back to a Woodgrove decision: governance requires Copilot Studio because of bank ALM and DLP; multi-channel reach requires Copilot Studio because the customer service experience must run in Teams *and* the customer portal; quick personal helpers (such as a relationship-manager email summarizer) are exactly where Agent Builder shines. Close with the explicit message: the rest of the workshop uses Copilot Studio because Woodgrove needs production-grade enterprise agents — Agent Builder is complementary.

### Module 04 — Custom Agent Design (slides 21-25, Labs 06)

- **Whiteboard: grounding source ladder.** Before Lab 06, draw the four-rung grounding ladder on a whiteboard or shared canvas: public web → SharePoint sites/libraries → uploaded files → Dataverse tables. Walk the room through each rung, explain the trust and governance trade-offs (public web is broadest but least controlled; Dataverse is narrowest but fully governed), and ask participants to identify which source type their own organization would use first. This sets the mental model for why Lab 06 starts with SharePoint grounding rather than a simpler but less realistic source.
- **Demo: SharePoint knowledge source metadata filters.** Open Copilot Studio, navigate to an existing agent's knowledge sources, and show the metadata filter configuration pane for a SharePoint knowledge source. Highlight how site-level versus library-level scoping affects answer quality and how metadata filters (content type, column values) narrow grounding to relevant documents. Participants who see this configuration before Lab 06 will understand the setup steps instead of clicking through them blindly.
- **Demo: model picker and selection trade-off.** Open the model picker in the agent settings and demonstrate switching from the "Default" label (GPT-4.1) to GPT-5 Chat. Explain the trade-off: GPT-5 Chat delivers stronger reasoning and richer completions but consumes more Copilot Credits per message; GPT-4.1 is faster and cheaper for high-volume, lower-complexity scenarios. Tell participants they will revisit model selection formally in Module 08 (Lab 17), but they should be aware of the picker location now so they can experiment during Lab 06 if they finish early.
- **Sidebar: Component Collections preview (≈5 min, concept only).** After Lab 06's SharePoint grounding step, spend five minutes on slide 30a to plant the cross-day callback. Frame the problem first: "If the **Customer Service Agent** and the **Loan Processing Agent** both need the latest **Woodgrove Product & Fee Disclosures**, do we duplicate the SharePoint setup in each agent?" Let the room answer no, then introduce Component Collections as Copilot Studio's reusable bundle (topics + knowledge + actions) owned by a primary agent and referenced by others. Promise the payoff in Lab 14. Do not click anything — Day 1 is capacity-constrained. Reinforce two FSI guardrails out loud: disclosures are Compliance-owned in production (not maker-owned), and **KYC is intentionally excluded** from this collection because KYC must remain a system-of-record connector lookup. Point participants to the `ComponentCollections` block in `workshop-config.example.json` as the source of truth for the collection name and source URLs.

### Module 05 — Topic Design (slides 26-33, Labs 07+08)

- **Orientation: Adaptive Cards designer.** Before Lab 08, open [adaptivecards.io/designer](https://adaptivecards.io/designer) in the browser and walk the room through the interface: the card payload editor on the left, the live preview in the center, and the host-app selector at the top. Load one built-in template (for example, the "Weather" or "Input Form" sample), point out the `TextBlock`, `Input.Text`, and `Action.Submit` elements, and show how editing JSON on the left updates the preview immediately. Participants who have never used the external designer need this 3-minute orientation so they can troubleshoot card JSON independently during Lab 08.
- **Demo: Power Fx formula mode and ForAll().** In Copilot Studio's topic editor, open a message node that uses an Adaptive Card, switch from the JSON view to Formula mode using the JSON-to-Formula toggle, and show the resulting Power Fx expression. Then write or paste one `ForAll()` expression that iterates over a table variable and builds a repeated card element (for example, `ForAll(CustomerAccounts, {title: ThisRecord.Name, value: ThisRecord.Status})`). Explain that Formula mode lets makers use familiar Power Fx instead of raw JSON, but the two views are interchangeable — edits in one appear in the other. Run the topic in a test session so participants see the rendered output before they attempt their own card in Lab 08.

### Module 06 — Actions and Events (slides 38-45, Labs 09-12)

- **Whiteboard: AI proposes, humans approve.** Before Lab 09 Part 2, draw a simple two-column whiteboard: "AI does" on the left (extract structured fields, summarize the application, route to the correct reviewer, write the audit trail back) and "Human does" on the right (read the summary, validate against the raw evidence, approve or reject, escalate to second tier above the policy threshold). This is the four-eyes / SR 11-7 boundary in one picture. Tell the room that *every* arrow crossing from left to right in a banking workflow is a governance moment that must be defended in a model-risk review.
- **Demo: Power Automate Approvals action setup.** Open Power Automate, create a throwaway flow, and add the **Start and wait for an approval** action. Show the four fields participants will fill in for Lab 09 Part 2: *Approval type* (`Approve/Reject – First to respond`), *Title*, *Assigned to* (must be a real human mailbox), and *Details* (where the AI summary goes). Trigger the flow once and show the resulting card in the **Approvals** app in Teams and the email in Outlook so participants know where to look during their own test runs.
- **Talk-through: SR 11-7 and OCC AI guidance in one minute.** Most participants have not read the supervisory letters. Give the room the 60-second version: SR 11-7 requires that any model used for material decisions has independent validation, ongoing monitoring, and a defined human-review point before customer impact. The OCC's 2024 AI/ML principles extend this to generative AI. The practical implication for Lab 09 Part 2 is that the agent and the AI summarization step are *decision support*, never the decision; the Approvals action is where human accountability lives. Mention **EU AI Act Article 14** (human oversight of high-risk AI systems) and **PRA SS1/23** as the equivalent UK / EU references.

### Module 07 — Lending Architecture (slides 46-54, Labs 13-15)

- **Set the GA frame up front.** Open Module 07 with one sentence: "Connected Agents went GA on November 30, 2025 — what we are about to build is production, not preview." This pre-empts questions from participants who saw older blog posts referring to a preview experience or extra opt-in. The toggle, picker, and Activity map shown in Lab 15 are the GA UX.
- **Walk the child-vs-connected matrix on slide 52a.** Spend 60–90 seconds on the matrix (lifecycle, reuse, knowledge/tools, versioning, best-for) before drawing any topology. Use the FSI framing from the Lab 15 README: a shared **Compliance Q&A** connected agent is the canonical reuse case banks recognize, because no one wants to re-implement disclosures and policy lookups inside every line-of-business agent. Document Review stays a **child** because it is tightly coupled to one loan-processing flow.
- **Whiteboard: orchestrator → child/connected agent topology.** Before Lab 15, draw the multi-agent topology on a whiteboard: a central orchestrator agent at the top, with arrows pointing down to one or more child or connected specialist agents. Label the orchestrator as the "Loan Processing Agent" and one child as a specialist (for example, "Document Review Agent"). Explain the routing: the orchestrator receives the user message, evaluates delegation instructions, and forwards to the appropriate specialist. The specialist completes its task and returns control to the orchestrator. Emphasize that each agent owns its own instructions, knowledge sources, and topic set — the orchestrator does not duplicate them.
- **Demo: "Let other agents connect" toggle.** Open a specialist agent's settings in Copilot Studio and show the "Let other agents connect to and use this one" toggle. Explain that enabling this toggle is what makes the agent available as a connected agent in the orchestrator's agent list. Without this toggle, the agent is standalone and invisible to other agents. Show where the toggle appears and what the confirmation message looks like. Note: with Connected Agents now GA, in-environment discovery works as soon as the agent is saved with the toggle on — no channel publish required.
- **Demo: delegation wiring in instructions.** Open the orchestrator agent's instructions pane and show one example of delegation wording — for example, `When the user asks about document review, delegate to the Document Review Agent.` Explain that the instruction text is what the orchestrator's language model uses to decide when to route; vague instructions produce unreliable delegation. Point out that the connected agent must already be toggled on and listed in the orchestrator's agent references before the instruction can reference it.
- **Pre-lab setup for the Lab 14 Component Collections extension (facilitator-only, optional).** Before Lab 14, pre-create the **Woodgrove Product & Fee Disclosures** Component Collection in the demo environment. Use the collection name from `ComponentCollections.ProductDisclosuresCollectionName` and seed it with the disclosure URLs in `ComponentCollections.DisclosureSourceUrls` from `workshop-config.example.json`. Bundle mortgage product fact sheets, personal loan disclosures, account/fee schedules, and the adverse action template references that Lab 21 produces. The participant extension is pure reference (~10 min) — they should not build the collection from scratch. If you skipped pre-creation, skip the extension; do not let it spill into Lab 15 time. When you do run it, deliver the cross-day payoff explicitly: "On Day 1 we previewed this; today the **Loan Processing Agent** references the same collection the **Customer Service Agent** uses, so Compliance-approved disclosure content stays consistent across channels." Reinforce both FSI guardrails: disclosures are Compliance-owned in production (the maker only references an approved collection), and **KYC is intentionally excluded** because KYC must stay a system-of-record connector lookup.

### Module 09 — Multimodal and Data (slides 63-72, Labs 19-21)

- **Walkthrough: Dataverse tables readiness check.** Before Lab 20, open the Power Apps maker portal, navigate to the Dataverse tables view, and verify that the Woodgrove Lending Hub tables are visible: `Loan Type`, `Assessment Criteria`, `Applicant`, `Financial Document`, and `Loan Application`. Walk the room through one table's columns and a few sample rows so participants know what correct data looks like. Emphasize that if any of these tables are missing or empty, the problem is data readiness (the Enterprise solution import or base data load did not complete), not a model or prompt problem. This single check prevents the most common Lab 20 support escalation.
- **Demo: completed Adverse Action Notice template and content control pattern.** Before Lab 21, open a completed `Adverse_Action_Notice_Template.docx` that contains Plain Text Content Controls and show it on screen. Point out the control titles (for example, `ApplicantName`, `ApplicationNumber`, `PrincipalReasons`, `CreditBureauName`, `ConsumerRightsStatement`), explain that these placeholders are replaced by Dataverse column values and prompt output at generation time, and show the resulting generated draft side by side with the template. Participants who see the before-and-after understand the pattern immediately and spend less time debugging field-name mismatches during Lab 21.
- **Frame the governance moment clearly.** Before participants start Lab 21, state the rule out loud: the agent produces a **draft Adverse Action Notice for the compliance officer to review** and must never send the notice to the applicant. Cite ECOA Regulation B (notice within 30 days of adverse action) and FCRA §615(a) (required disclosures when a consumer report is used) so the room understands why the lab is structured around a `DRAFT-` filename, a chat-reply review reminder, and a `Denied` guard branch in the flow. Remind the room that the workshop wording is a simplified, illustrative draft, not legal-grade text — banks must use their own legal-approved templates with qualified compliance and legal review.

### Module 10 — MCP and Extensibility (slides 73-76, Lab 22)

- **Whiteboard: classic connectors vs MCP mental model.** Draw two diagrams side by side. On the left, draw the classic connector model: one connector corresponds to one action, and the agent must be wired to each action individually (for example, "Get items from SharePoint list" is one connector action, "Send email" is another). On the right, draw the MCP model: one MCP server exposes many tools, and the agent discovers and invokes them dynamically through a single server connection. Label the MCP side with the Work IQ MCP server as the concrete example. Explain that the MCP model reduces per-action wiring but requires the server to be registered, consented, and governed — the agent cannot call arbitrary endpoints.
- **Live demo: MCP server catalog → connection → consent flow.** Before Lab 22, walk through the full MCP onboarding sequence live on screen: open Copilot Studio → navigate to the MCP server catalog → select the Work IQ server → create the connection → walk through the consent card that appears (explain what permissions are being granted and why) → confirm the server appears in the agent's tool list. Participants who see this end-to-end flow before Lab 22 will recognize each step when they perform it and will not stall at the consent card, which is the most common point of confusion.
- **Forward-guidance callout: transport for custom MCP servers.** The lab itself uses pre-built Microsoft-hosted Work IQ servers added through the in-product wizard, so participants never touch transport settings. If a participant asks about building their own MCP server later, give the one-line answer: **Streamable HTTP** is the recommended transport for new MCP servers in Copilot Studio, and **Server-Sent Events (SSE)** is deprecated for new servers (existing SSE-based servers continue to work but should be migrated). Do not detour into custom-server setup during the workshop.

- **Regulatory anchors for Slide 101.** Cite the five anchors specifically and tie each to a Three-Zones or ALM practice already on the earlier slides:

### Module 12 — Evaluation and ROI (slides 81-85, Lab 24)

Copilot Studio Evaluation has been generally available since April 2026. Frame this module as a release-gate practice rather than a preview, and lean on the new GA capabilities (multi-grader, multi-turn, auto-generated test inputs) as the default shape of a healthy test set.

- **Live demo: Evaluation tab orientation.** Before Lab 24, open the **Evaluation** tab in Copilot Studio and walk the room through the three main areas: the **Test sets** list (where CSV-based test cases are uploaded or auto-generated from agent instructions), the **Graders** list (where assessment criteria are configured, including multi-grader stacks per case), and the **Run results** panel (where completed evaluation runs show pass/fail counts, per-grader pass rates, and individual case outcomes). Point out the "New evaluation run" button and explain the workflow: create or import test set → attach graders → run → review results. Participants who see this layout before Lab 24 will orient themselves immediately instead of searching for the right tab.
- **Explanation: grader types and multi-grader.** Briefly describe the seven GA grader types so participants understand what each one measures before they configure their own:
  - **General response quality** — assesses whether the agent's response is helpful, accurate, and well-formed relative to the test case's expected answer.
  - **Semantic meaning** — checks whether the agent's response preserves the meaning of the expected answer, even if the wording differs.
  - **Capability usage** — verifies that the agent invoked the correct tools (actions, connectors, MCP servers) during the conversation, not just that the final text was correct.
  - **Keyword presence** — confirms that specific required keywords or phrases appear in the agent's response, useful for compliance or branding checks.
  - **Text similarity** — measures how closely the agent's response text matches the expected answer at a string level.
  - **Exact match** — checks whether the agent's response is an exact match to the expected answer.
  - **Custom Graders** — a Classification method that lets organizations encode their own policies, quality standards, or rules directly into the evaluation.
  Then call out **multi-grader**: at GA you can attach more than one grader to a single test case, so one run scores quality, tool use, and compliance keywords together. This is the right default pattern for regulated lending responses.
- **Explanation: Zones of Coverage and multi-turn.** Use the PowerCAT **Zones of Coverage** framing — **Capability**, **Regression**, **Safety** — to keep the test set balanced. Demonstrate **multi-turn evaluation** by walking a two-turn case where an ambiguous loan-officer prompt should trigger a clarifying question first; show how the run scores the full conversation, not just the last message. Mention **auto-generated test inputs** as a draft accelerator (Copilot Studio drafts cases from the agent instructions) but stress that every generated case must be edited to fit the lending scenario before it earns a place in the release-gate set.
- **Demo: Activity map on a failed test case.** After showing the evaluation tab, open one failed test case from a previous run (or trigger one intentionally) and click **Show activity map**. Walk through the activity map view: show the sequence of nodes the agent traversed, highlight where the conversation diverged from the expected path, and explain how the map helps diagnose whether the failure was a grounding problem (wrong knowledge source), a routing problem (wrong topic or delegation), or a generation problem (correct data but poor completion). This diagnostic view is the single most useful tool for improving evaluation scores after Lab 24.
- **Awareness mention: EvalGate CI/CD (advanced, take-home).** For IT Pro and Developer audiences, mention the **EvalGate** pattern (e.g., the **EvalGateADO** Azure DevOps integration) as a way to fail a build when the evaluation pass rate drops below a threshold. Keep this as awareness only — do not demo or run it in class. It is the natural next step for teams that already have a build pipeline and want evaluation in their release gate.

> **REGULATORY GUARDRAILS (read before opening the maker portal).** Cite these by name during the demo; do not paraphrase. **SR 11-7 (Federal Reserve)** — model risk management, effective challenge, human accountability over models that influence credit decisions. **ECOA / Regulation B §1002.9** — adverse action notices require specific principal reasons and a documented human-reviewed process. **OCC Bulletin 2021-39 / OCC AI guidance** — apply existing model risk, third-party risk, and governance frameworks to AI with human oversight commensurate with risk. **EU AI Act Annex III §5(b)** — credit-scoring AI is high-risk and requires human oversight, risk management, logging, transparency, and post-market monitoring. **Four-eyes / two-person review** — material credit and operational actions require two independent reviewers; an autonomous agent acting alone on a credit decision violates this on its face. The Tier-2 Triage Assistant stays on the right side of every one of these because it never decides, never communicates externally, and never closes the loop without a human.

- **Live walkthrough: trigger setup in the maker portal (~5 min).** Open the **Loan Processing Agent** in the facilitator demo environment, open the **Triggers** view, and show the schedule (or `On record created` for high-priority Service Requests). Walk the room through the four boundary checks they should look for in any autonomous topic: *no outbound channel*, *no Dataverse write to the borrower record*, *no decision row written to the credit decisioning table*, *no third-party API call*. Make the boundary checks the headline, not a footnote.
- **Live walkthrough: the memo (~10 min).** Open the SharePoint document library at `Autonomous.MemoLibraryUrl` and open a `triage-memo-*.md` from a prior demo run (do not generate one live unless the room asks). Read the memo's **"Items the agent declined to recommend"** section and **"Audit"** section aloud. These two sections are the demo's most important teaching moments — the agent declined to score a credit-decision-shaped Service Request, and the audit block records that no external messages were sent and no Dataverse write occurred to any loan-application or credit-decision row.
- **Live walkthrough: activity / transcript view (~5 min).** Open the unified activity and transcript view for the most recent autonomous run and show that every Dataverse query and the memo write are logged, while there is no outbound message or credit-decision write. This is the *evidence trail* a model risk manager would request under SR 11-7 effective challenge.
- **Discussion prompt (~10 min).** Pose: *"What would have to be true before you would let this same trigger deliver its memo to the customer instead of to your triage manager? List the controls."* Let small groups answer first. Capture the controls (explicit human review, adverse-action workflow, model validation and challenger model, transcript retention and supervisory review, fair-lending impact testing, complaint handling, opt-in/opt-out for digital communication) on the whiteboard. The point is to make participants articulate the long list that separates "internal triage memo" from "customer-facing decision."
- **Anti-patterns to call out explicitly.** Read the *"What we are NOT doing"* list from `autonomous-triage-demo.md` aloud, do not soften it: no credit decisions, no adverse-action notices, no borrower contact, no Dataverse writes to applications or decision rows, no external API calls, no replacement of the four-eyes principle, and no claim of SR 11-7 compliance for the agent itself.
- **Cleanup callout (end the demo with this on screen).** The autonomous trigger continues to fire after the workshop and continues to consume Copilot Credits in the facilitator demo environment. Run `pwsh -File .\workshop\automation\Disable-WorkshopAutonomousTriggers.ps1 -ListOnly` first, then the same command without `-ListOnly` to disable. The script refuses to run unless `Workshop.EnvironmentPurpose` identifies the target as a facilitator demo or fallback environment, so it cannot accidentally touch a student environment. Add this to the closeout checklist for the day.
