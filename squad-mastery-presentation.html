Section 01

Why Squad?

Understanding the problem Squad solves and the paradigm shift it represents.

Limitations of Single-Agent Copilot Workflows

Traditional AI-assisted development relies on a single agent responding to prompts. This approach hits fundamental walls: limited context windows that can't hold an entire codebase, no persistent memory between sessions, no specialization (one agent tries to be expert at everything), and no built-in accountability or review process.

💭

Context Limits

A single agent loses track of decisions made yesterday. Every session starts from scratch.

🎯

No Specialization

One agent can't be an expert frontend dev, backend architect, tester, and tech writer simultaneously.

⏳

Sequential Bottleneck

One agent works on one thing at a time. Complex features require serial execution.

🚫

No Governance

No review gates, no lockout on bad code, no escalation paths. Trust is implicit, not enforced.

The Need for Multi-Agent Orchestration

Real software teams have specialists: frontend developers, backend engineers, testers, technical writers. Each brings domain expertise and a unique perspective. Squad brings this model to AI-assisted development — a team of specialized agents that work in parallel, remember decisions, and operate under governance rules.

Paradigm Shift

Squad moves you from prompting (telling one AI what to do) to coordinating (directing a team of specialists and reviewing their work). You become the technical lead, not the typist.

Section 02

Core Concepts

The foundational ideas that make Squad work.

🏗

What Squad Is

An open-source framework for human-directed, multi-agent AI development teams. Agents live in your repo as .squad/ and grow smarter with every session.

🐙

Repo-Native Agent Teams

Your team state — roster, decisions, skills, memory — lives in .squad/. Commit it, clone it, share it. Team knowledge travels with your code.

📡

Coordinator + Specialists

A routing engine (coordinator) decomposes your requests and spawns specialized agents in parallel. Each agent has a charter defining their role and expertise.

🧠

Persistent Memory

Agents remember decisions, conventions, and architecture patterns across sessions. Personal histories + shared decisions = a team that gets smarter over time.

🛡

Human-in-the-Loop Governance

File-write guards, PII scrubbing, reviewer lockout, and clear escalation points. Rules are enforced in code, not suggestions in prompts.

⚙

SDK-First Design

Define your team in TypeScript with squad.config.ts. Typed, testable, version-controlled configuration with full autocomplete.

Key Components

📡 Coordinator

▶

The coordinator is Squad's routing engine. It reads your request, checks routing rules in .squad/routing.md, and decides which agents to spawn. If you say "team," it decomposes the work and launches multiple agents in parallel. If you name an agent, it routes directly to them.

The coordinator also handles dependency analysis, deadlock detection, and reviewer gates. It never does the work itself — it orchestrates who does.

🤖 Specialized Agents

▶

Each agent is a specialist with a charter, role, and persistent memory. Agents are spawned as independent subprocesses with their own context windows and tools. They read .squad/decisions.md and their own history before working, then write results back.

Agents never see each other's conversations — the coordinator orchestrates all coordination through shared files.

RoleWhen IncludedResponsibility
LeadAlwaysTriages, reviews, unblocks
Core DevAlwaysMain implementation
TesterIf tests existTest writing and validation
FrontendIf React/Vue/Angular detectedUI implementation
BackendIf API/DB detectedServer-side logic
DevRelIf docs presentDocumentation
ScribeAlways (silent)Decision logging

│   └── ...

│   └── ...

├── ceremonies.md              # Team ceremonies
├── casting-state.json         # Agent names, universe theme
└── model-config.json          # Per-agent model overrides

📋 Scribe & Ralph

▶

Scribe is a silent agent that tracks decisions and logs sessions. Every team has a Scribe. You never talk to them directly — they work in the background, merging decisions from all agents into .squad/decisions.md. They also handle deduplication and conflict detection.

Ralph is the work monitor. He watches your GitHub or GitLab issues, tracks work in progress, and alerts the team when something is ready. He's silent unless you ask for status.

🔀 Routing & Task Orchestration

▶

Routing uses three strategies (first match wins):

StrategyHow It WorksExample
NamedYou say who does it"Fenster, fix the login bug"
DomainPattern matching in routing.mdsrc/api/** → Backend
Skill-awareCapability check in skills/Auth expertise → Backend

If nothing matches, the Lead triages. GitHub issues with squad:{member} labels route directly to that agent.

Section 04

Installation & Setup

Three commands to a full team.

Step 1: Install the CLI

Terminal
# Global install (recommended)
npm install -g @bradygaster/squad-cli

# Or one-off with npx (no install needed)
npx @bradygaster/squad-cli init

# Verify installation
squad --version

Step 2: Initialize Your Team

Terminal
cd your-project
squad init

The init flow walks through four steps:

StepWhat Happens
1. DiscoverySquad scans your repo: languages, file structure, test frameworks, dependencies
2. ProposalSuggests a roster with 3–7 members tailored to your stack
3. ConfirmationYou review and customize: accept, add roles, remove roles, rename
4. CreationWrites .squad/ directory, creates charters, sets up coordinator

What Gets Created

.github/agents/squad.agent.md   ← Coordinator agent file
.squad/                         ← Team state directory
  ├── team.md                   ← Your roster
  ├── routing.md                ← Work assignment rules
  ├── decisions.md              ← Shared memory
  └── agents/                   ← Individual agent charters

Step 3: Start Working

Terminal
# Launch the interactive shell
copilot --agent squad

# Or address your team directly
squad > Team, build the recipe listing page
First Run Experience

On first run, Squad uses a CastingEngine to assign agent names from fictional universes (The Usual Suspects, Ocean's Eleven, etc.). Each agent gets a personality, backstory, and voice. Use squad init --roles for plain role names instead.

Optional: Typed Configuration

squad.config.ts
import { defineConfig } from '@bradygaster/squad-sdk';

export default defineConfig({
  team: {
    name: 'my-squad',
    root: '.squad',
    description: 'My project team',
  },
});
Section 05

How Squad Works: Execution Model

From task input to delivered results.

The Execution Flow

💬 "Team, implement user auth"
↓
🔍 Coordinator: Dependency Analysis
↓ No dependencies found
🚀 FAN-OUT: 4 agents spawned in parallel
↓
Backend: API endpoints
Frontend: Login form
Tester: Integration tests
DevRel: Auth docs
↓
📥 COLLECT: Aggregate results
↓
✅ Labeled results returned to you

Parallel vs Sequential Execution

ModeWhen UsedBehavior
Background (parallel)Independent work, no data dependenciesAgents run simultaneously, coordinator polls for completion
Sync (sequential)Data dependency or reviewer gateAgent runs, coordinator waits before proceeding
Eager Execution Philosophy

Squad's default is eager parallelism — launch everything that can run, let the coordinator handle synchronization. Default concurrency: 5 agents in parallel. Adjustable with: "Run at most 2 agents at once to save costs"

Response Modes

Squad auto-selects the right level of effort for each request:

ModeTimeTriggered By
Direct~2–3sStatus checks, factual questions
Lightweight~8–12sSmall fixes, typos, quick follow-ups
Standard~25–35sNormal work requests (named agent)
Full~40–60sComplex multi-domain tasks ("Team, ...")

Feedback Loop: Reviewer Protocol

When a reviewer (Lead or Tester) rejects work, the original agent is locked out from self-revision. This prevents endless fix-retry loops and forces fresh perspective or human intervention.

Agent A writes code → Lead rejects → Agent A LOCKED OUT
  → Coordinator reassigns to Agent B
  → OR escalates to you
Governance Enforced

Lockout is task-specific (not a global ban), session-persistent (survives restarts), and clearable by you ("Unlock Fenster for issue #42"). If all agents are locked out, the coordinator escalates to you with options.

Section 06

Features Breakdown

Detailed exploration of each major capability.

Core

Multi-Agent Orchestration

Squad launches independent work in parallel by default. The coordinator performs dependency analysis, fans out all independent agents simultaneously, waits for results, then collects and synthesizes. No sequential bottlenecks unless data dependencies or reviewer gates require them.

Why it matters: Faster throughput, better resource utilization, and resilience to blocking. If one agent stalls, others keep working.

Team, build the new dashboard: API endpoints, React components, tests, and docs
Memory

Persistent Memory

Squad's memory is layered: Personal Memory (agents/{name}/history.md) stores agent-specific learnings that only that agent reads. Shared Memory (decisions.md) stores team-wide decisions that every agent reads before working. Memory compounds over time — after a few sessions, agents stop asking questions they've already answered.

Why it matters: No more repeating context. Your team gets smarter with every session.

What decisions has the team made about testing strategy?
StageWhat Agents Know
First sessionProject description, tech stack, your name
After a few sessionsConventions, component patterns, API design
Mature projectFull architecture, tech debt map, regression patterns
Identity

Agent Charters

Each agent has a charter.md that defines their identity, expertise, tools, and voice. The charter is compiled into the system prompt at spawn time. You can edit charters directly or ask the coordinator to adjust them.

Why it matters: Charters give agents consistent personality and domain focus across sessions.

Change the tester to focus on integration tests instead of unit tests
Orchestration

Routing & Coordination

The coordinator uses a three-layer routing strategy: Named routing (you say who), Domain routing (pattern matching on file paths), and Skill-aware routing (capability check). First match wins. If nothing matches, the Lead triages.

Why it matters: Work always goes to the right specialist without manual assignment.

Fenster, implement the new search API. Hockney, write integration tests for it.
Memory

Scribe & Decision Tracking

The Scribe is a silent agent that periodically reads all entries from .squad/decisions/inbox/, merges them into the canonical decisions.md, deduplicates overlapping decisions, and propagates updates to affected agents. Decisions are captured from agent work, user directives, and retrospectives.

Why it matters: Decisions are never lost. The team's shared brain grows automatically.

Always use single quotes in TypeScript
Knowledge

Skills System

Skills are reusable knowledge files at .squad/skills/{skill-name}/SKILL.md. Squad ships with 8 built-in skills (error-recovery, secret-handling, git-workflow, etc.) and agents earn new skills from real work. Skills have a confidence lifecycle: Low → Medium → High.

Why it matters: Skills compound over time and are portable across projects. A trained team carries their expertise to new repos.

Create a skill for our deployment process
Governance

Directives

Directives are team rules that persist across sessions. When you say "always", "never", "from now on", or "remember to", Squad captures it and writes it to permanent memory. Every agent reads these before working.

Why it matters: Shape team behavior with natural language. Rules are enforced consistently.

Never commit directly to main. Always run tests before pushing.
Process

Ceremonies

Structured team meetings that happen at key moments. Design Reviews auto-trigger before multi-agent tasks involving 2+ agents modifying shared systems. Retrospectives auto-trigger after build failures or reviewer rejections. Both can be triggered manually or disabled.

Why it matters: Prevents agents from building conflicting implementations. Retros produce decisions that improve future work.

Run a design review before we start the authentication rebuild
Extensibility

Plugins & MCP Integration

Plugins are community-curated bundles of agent templates, skills, and best practices. Install them to give your squad instant expertise (React, Azure, security, etc.). MCP (Model Context Protocol) bridges agents and external services — GitHub, Trello, notifications, deployments. Agents discover and use MCP tools automatically.

Why it matters: Extend Squad to fit any workflow without building from scratch.

Install the AWS deployment plugin. Configure the GitHub MCP server.
Safety

Governance Model

Squad enforces governance through multiple mechanisms: file-write guards prevent unauthorized changes, PII scrubbing (squad scrub-emails) removes sensitive data, the reviewer lockout protocol prevents endless retry loops, and clear escalation points ensure humans intervene when needed. Trust levels for PRs range from Full Review (default) to Self-Managing (solo projects only).

Why it matters: Responsible AI at agentic speed. Humans stay accountable.

Lead, review the PR from Fenster. Check for security issues.

Section 08

Advanced Patterns

Scaling, multi-repo, conflict handling, and continuous workflows.

🌐 Multi-Repo & Multi-Squad Usage

▶

Squad supports multiple deployment topologies:

PatternHowUse Case
Personal Squadsquad init --globalSame agents across all your projects
Remote Initsquad init --mode remote ../team-repoShared team identity for monorepos
Upstream Inheritancesquad upstream addInherit from an org-level squad definition
Export/Importsquad export / squad importMove a trained team to a new repo

In dual-root mode, project-specific state lives in your local .squad/ while team identity (casting, charters, shared decisions) lives in the remote location.

📈 Scaling Execution

▶

Squad provides multiple scaling mechanisms:

  • Concurrency limits: "Run at most 3 agents at once" — batches work in groups
  • Economy mode: Falls back to cheaper models when rate limits approach
  • squad watch --max-concurrent N: Control parallel issues per round
  • KEDA Autoscaling: Kubernetes-based autoscaling for enterprise deployments
  • Distributed Mesh: Multiple Squad instances coordinating across services
Run at most 2 agents at once to save costs
Switch to economy mode

⚠ Agent Conflict Handling

▶

Circular Dependencies: When Agent A needs output from Agent B and vice versa, the coordinator detects the cycle and prompts you to choose resolution (run A first, run B first, or redesign).

Directive Conflicts: When a new directive contradicts an existing one, the Scribe detects the semantic overlap and asks you to replace, merge, or skip.

Deadlock: If all capable agents are locked out for a task, the coordinator escalates with options: manual fix, unlock an agent with guidance, or close as won't-fix.

🛡 Governance & Safety Patterns

▶

Squad's governance model aligns with enterprise requirements:

MechanismWhat It Does
File-write guardsPrevent unauthorized changes to protected paths
PII scrubbingsquad scrub-emails removes email addresses from state files
Reviewer lockoutRejected agents cannot self-revise; forces fresh perspective
Escalation pointsClear paths to human intervention when agents are stuck
Trust levelsFull Review → Selective Review → Self-Managing (configurable per repo)
Audit loggingAll decisions and lockouts logged in orchestration-log/

For regulated industries (FSI), these patterns map to governance frameworks like the FSI Agent Governance Framework, which defines 78 controls across security, management, reporting, and data pillars.

🔄 Continuous Workflows (Looping)

▶

The squad loop command runs a prompt-driven work loop from a loop.md file. Each cycle, Loop sends your prompt to Copilot and loops again at your chosen interval.

loop.md
---
configured: true
interval: 10
timeout: 20
description: "Monitor failing CI and fix issues"
---
# CI Monitor Loop
Each cycle, you will:
1. Check GitHub Actions for failures in main branch
2. If failures exist, investigate the top 1-2
3. If fixable, create a PR with the fix
4. Report findings
squad loop                    # Run the loop
squad loop --init             # Create a starter loop.md
squad loop --interval 3       # Override interval

When NOT to Use Squad

📝

Simple Scripts

A 50-line utility script doesn't need a team. Use a single Copilot prompt.

💬

One-Off Questions

If you just need a quick answer or code snippet, Squad's overhead isn't justified.

🔒

Non-Code Workflows

Squad is designed for software development. For document writing or data analysis, other tools are better suited.

Section 10

Teaching Mode

How to explain Squad to others and run effective demos.

The One-Liner

Elevator Pitch

"Squad is like hiring a dev team that lives in your repo. You describe what you're building, and specialized AI agents fan out to work in parallel — with persistent memory, built-in governance, and you as the technical lead."

Common Misconceptions

MisconceptionReality
"It writes code for me automatically"Squad works with you. You direct, review, and decide. Agents propose; you approve.
"Agents talk to each other"Agents never see each other's conversations. They communicate through shared files (decisions.md) and the coordinator orchestrates.
"It replaces my team"Squad makes individuals more productive. It extends human teams, not replaces them.
"It's just fancy prompting"Squad has persistent memory, governance enforcement, parallel execution, and a full lifecycle. It's a framework, not a prompt template.
"I need cloud infrastructure"Squad runs locally. No servers, no databases. Just .squad/ in your repo.

Demo Script (5 Minutes)

1. Show the problem (30s): Open a complex project. Show how a single Copilot prompt struggles with multi-file changes.
2. Init Squad (60s): Run squad init. Show the team proposal. Accept it. Show the .squad/ directory.
3. Parallel execution (90s): Say "Team, build the recipe listing page". Show agents fanning out. Point out the orchestration log.
4. Memory in action (60s): Set a directive: "Always use single quotes". Show it in decisions.md. Ask the team a question about past decisions.
5. Governance (60s): Show a reviewer rejection. Explain lockout. Show the escalation path. Emphasize: "Humans stay in charge."

Key Talking Points for Technical Audiences

For Developers

Focus on: parallel execution, persistent memory, SDK-first design, TypeScript config, and the fact that .squad/ is just Markdown files you can edit.

For Engineering Managers

Focus on: governance model, reviewer protocol, ceremonies (design reviews/retros), audit logging, and trust levels for PR management.

For Security/Compliance

Focus on: file-write guards, PII scrubbing, reviewer lockout, escalation paths, audit trails, and alignment with governance frameworks.
